# CSF Expansion Engine - Projection Runtime Proposals

## 概述

本文档基于 prompt.txt 中的指导思想，提出将当前系统从 "structure-aware analyzer" 转换为 "working-set-aware projection runtime" 的技术提案。

当前系统问题：
- 虽然有 projection 概念，但实际上只是把结构换一种格式打印出来
- 没有真正控制 working set
- Cognitive Matrix 被当作 metadata，而不是驱动 projection runtime

目标架构：
```
Structural Truth Layer
    ↓
Canonical IR
    ↓
Recognition Graph
    ↓
RPCM Scheduler
    ↓
Stable Projection View
```

## 提案一：Projection Scheduler 核心层

### 1.1 问题背景

当前系统流程：
```text
AST → CSF → projection → dump
```

问题：
- projection 是 static 的，不是 working-set-reactive 的
- 系统输出所有 state、mutation、causal effect、fanout
- working set 直接爆炸，用户无法处理
- Cognitive Matrix 只用于 observation，不用于 action

### 1.2 第一轮提案：WS Budget 控制

**核心思想**：为每个 Recognition Region 设置预算，超过自动 compress。

**技术方案**：
```python
class WSBudget:
    MAX_R0 = 12  # 最大结构节点数
    MAX_R1_CHAIN = 5  # 最大执行链长度
    MAX_R2_PROPAGATION = 4  # 最大传播深度

    def check_budget(self, ws_state: dict) -> dict:
        """检查 WS 预算并返回压缩建议"""
        violations = []

        if ws_state['R0']['node_count'] > self.MAX_R0:
            violations.append({
                'region': 'R0',
                'current': ws_state['R0']['node_count'],
                'limit': self.MAX_R0,
                'action': 'compress_low_priority_nodes'
            })

        if ws_state['R1']['chain_length'] > self.MAX_R1_CHAIN:
            violations.append({
                'region': 'R1',
                'current': ws_state['R1']['chain_length'],
                'limit': self.MAX_R1_CHAIN,
                'action': 'split_execution_chain'
            })

        if ws_state['R2']['propagation_depth'] > self.MAX_R2_PROPAGATION:
            violations.append({
                'region': 'R2',
                'current': ws_state['R2']['propagation_depth'],
                'limit': self.MAX_R2_PROPAGATION,
                'action': 'compress_propagation_region'
            })

        return violations
```

**优势**：
- 实现简单，直接控制认知负载
- 可配置预算参数
- 自动触发压缩

**劣势**：
- 固定预算可能不适合所有场景
- 需要定义优先级规则

**技术成熟度**：高（常见资源管理模式）

**自我评估**：
- **服务项目目标**：✓ 是。核心功能
- **优先级**：高。必须实现

## 提案二：Focus Lock 机制

### 2.1 问题背景

用户在编辑时 focus 某个函数，但系统仍然显示所有无关分支，导致认知负载过高。

### 2.2 第一轮提案：Focus Lock 冻结无关分支

**核心思想**：用户 focus 时，自动冻结无关分支。

**技术方案**：
```python
class FocusLock:
    def lock_focus(self, csf: dict, focus_node_id: str) -> dict:
        """锁定 focus，冻结无关分支"""
        focus_node = csf['nodes'][focus_node_id]

        # 计算相关节点（focus 节点的依赖链和被依赖链）
        related_nodes = self._compute_related_nodes(csf, focus_node_id)

        # 冻结无关节点
        frozen_nodes = []
        for node_id, node in csf['nodes'].items():
            if node_id not in related_nodes:
                node['frozen'] = True
                frozen_nodes.append(node_id)

        return {
            'focus_node': focus_node_id,
            'related_count': len(related_nodes),
            'frozen_count': len(frozen_nodes)
        }

    def _compute_related_nodes(self, csf: dict, focus_node_id: str) -> set:
        """计算相关节点"""
        related = set()

        # 上游依赖
        related.update(self._get_upstream_dependencies(csf, focus_node_id))

        # 下游影响
        related.update(self._get_downstream_affects(csf, focus_node_id))

        # 同级兄弟节点
        related.update(self._get_siblings(csf, focus_node_id))

        return related
```

**优势**：
- 实现简单
- 直接减少认知负载
- 符合用户直觉

**劣势**：
- 可能过度隐藏

**技术成熟度**：高（常见 IDE 功能）

**自我评估**：
- **服务项目目标**：✓ 是。核心功能
- **优先级**：高。必须实现
- **风险**：低

## 提案四：Adaptive Projection 自适应投影

### 4.1 问题背景

当前 projection 是 static 的，不能根据用户目标、WS 预算、当前 focus、thrash history 动态调整。

### 4.2 第一轮提案：Dynamic Projection Scheduler

**核心思想**：实现真正的 projection scheduling runtime。

**技术方案**：
```python
class ProjectionScheduler:
    def schedule_projection(
        self,
        csf: dict,
        user_goal: str,
        ws_budget: WSBudget,
        current_focus: str,
        thrash_history: list
    ) -> dict:
        """调度 projection"""
        # 1. 检测 projection mode
        mode = self._detect_mode(user_goal)

        # 2. 应用 mode 过滤
        filtered_csf = self._apply_mode(csf, mode)

        # 3. 检查 WS 预算
        violations = ws_budget.check_budget(self._compute_ws_state(filtered_csf))

        # 4. 如果违反预算，压缩
        if violations:
            filtered_csf = self._compress(filtered_csf, violations)

        # 5. 如果有 focus，锁定
        if current_focus:
            filtered_csf = self._lock_focus(filtered_csf, current_focus)

        # 6. 如果检测到 thrash，regroup
        if thrash_history:
            filtered_csf = self._regroup(filtered_csf, thrash_history)

        # 7. 渐进式揭示
        projection = self._progressive_reveal(filtered_csf)

        return {
            'mode': mode,
            'projection': projection,
            'ws_state': self._compute_ws_state(projection),
            'compressed_count': len(csf['nodes']) - len(projection['nodes']),
            'thrash_risk': self._compute_thrash_risk(thrash_history)
        }
```

**优势**：
- 真正实现 working-set-aware projection
- 动态调整认知可见性
- 符合理论目标

**劣势**：
- 实现复杂度高
- 需要调优多个参数
- 依赖多个子模块

**技术成熟度**：低（需要大量工程优化）

**自我评估**：
- **服务项目目标**：✓ 是。核心目标
- **优先级**：高。最终目标

### 5.3 业界参考方案

#### 5.3.1 VS Code 模块化实践

VS Code 采用分层架构：
- `src/vs/base/` - 基础层
- `src/vs/platform/` - 平台层
- `src/vs/workbench/` - 工作台层
- `src/vs/editor/` - 编辑器层

每个层级按功能进一步细分：
- `services/` - 服务层
- `common/` - 通用组件
- `browser/` - 浏览器相关

#### 5.3.2 PyTorch 模块化实践

PyTorch 采用功能模块化：
- `torch/nn/` - 神经网络模块
- `torch/optim/` - 优化器模块
- `torch/utils/` - 工具模块
- `torch/distributed/` - 分布式模块

每个模块内部按具体功能细分：
- `functional/` - 函数式 API
- `modules/` - 模块化组件
- `loss/` - 损失函数

#### 5.3.3 推荐的模块化原则

1. **单一职责原则**：每个模块只负责一个功能
2. **依赖倒置原则**：高层模块不依赖低层模块
3. **接口隔离原则**：模块间通过接口通信
4. **开闭原则**：对扩展开放，对修改关闭

### 5.5 Pattern 类型扩展

#### 5.5.1 当前 Pattern 类型

当前系统已实现的 Pattern：
- `pipeline` - 流水线模式
- `branch` - 分支模式
- `fetch_transform_cache` - 获取-转换-缓存模式
- `validation_chain` - 验证链模式
- `event_handler` - 事件处理模式

#### 5.5.2 建议新增 Pattern 类型

1. **Singleton Pattern（单例模式）**
   - 识别：类中只有一个实例
   - 规则：类有私有构造函数，有静态获取实例方法

2. **Factory Pattern（工厂模式）**
   - 识别：创建对象的工厂类
   - 规则：有创建方法，返回不同类型的对象

3. **Observer Pattern（观察者模式）**
   - 识别：事件订阅和通知机制
   - 规则：有订阅方法，有通知方法

4. **Strategy Pattern（策略模式）**
   - 识别：可互换的算法族
   - 规则：有策略接口，有多个实现

5. **Decorator Pattern（装饰器模式）**
   - 识别：动态添加功能
   - 规则：有装饰器函数或类

6. **Repository Pattern（仓储模式）**
   - 识别：数据访问抽象层
   - 规则：有 CRUD 操作，有数据映射

7. **Service Layer Pattern（服务层模式）**
   - 识别：业务逻辑层
   - 规则：有业务方法，不直接操作数据库

8. **Command Pattern（命令模式）**
   - 识别：命令封装
   - 规则：有执行方法，有撤销方法
