核心问题Core Problem
改动成本高的根源：state 的访问契约是隐式的，代码结构和认知结构不一致。
两个目标缺一不可：让「谁能改 / 谁要用」对每个 state 显式可查，同时让用户在整理过的虚拟层操作，不直接面对 spaghetti。The root cause of high change cost: state access contracts are implicit, and code structure misaligns with mental models.
Two inseparable goals: make "who can write / who depends on it" explicitly queryable for every state, and let users operate on a clean virtual layer instead of facing spaghetti directly.
边界定义Boundaries
这个系统做什么What this system does
  • 把隐式的 write sites / use sites（def-use chain）提取出来，让改动上下文最小
  • 提供虚拟层作为编辑界面，改动 patch 回真实层，不直接改 spaghetti
  • 真实层被外部修改时，全量重新解析，虚拟层重新生成
  • 改动前预告 readers 影响范围，不需要运行
这个系统不做什么What this system does NOT do
  • 不是独立 spec 文件系统（契约活在代码里）
  • 不是自由代码生成器（生成层受 writers 约束，且是可选的最后一层）
  • 不用 assert 描述动态系统（不是 test framework）
  • 不强求全局理解（局部问题 ≤ 1 view）

核心数据结构 · 访问契约Core Data Structure · Def-Use Contract
系统一切分析的基础。学术上对应 def-use chain（编译器理论，1960s），工程上对应 data flow / taint analysis（SonarQube、CodeQL 都有实现）。
与现有工具的区别：现有工具的目标是安全漏洞检测（source → sink 危险路径）；这里的目标是认知负担最小化（哪些 write sites 需要被人理解和修改）。角度不同，直接复用 Joern 的 def-use 提取，不自己写扫描逻辑。
source 字段区分工具推断（inferred）和人工覆盖（explicit）。
字段含义用途
state 名称 + 定义位置 找 state 去哪找
writers 赋值操作所在的函数/模块 改动只需要看这里
readers 引用该 state 的函数/模块 改了之后影响谁
source inferred / explicit inferred = 工具推断可被覆盖；explicit = 人工声明强制验证

进阶行为 · 隔离策略Advanced · Isolation Strategy
虚拟层的存在让物理隔离成为默认状态，不需要人声明 capsule。
隔离策略主要用于真实层的重构引导：当 writers 过多时，给出升级路径。
主攻：访问隔离（writers 收敛）

目标：让每个 state 的 writers 列表尽可能短。writers 越少，改动上下文越小。
工具检测 writers 数量，超过阈值时提示重构。

  • inferred 契约自动推断，writers 数量是健康度指标
  • explicit 契约人工声明后，工具强制验证，额外 writer 实时标红
  • writers = 1 是最理想状态，改动上下文缩到最小
技术预案：时序隔离 + 物理隔离

触发条件：某个 state 的 writers ≥ 3，且业务上无法合并。

writers ≥ 3 直接写同一 state ↓ 升级：改为发消息到 event bus / channel state 只有一个消费者处理所有写请求 ↓ 自然结果：物理边界清晰，writers = 1 虚拟层自动反映新结构，改动上下文收敛
  • event bus 本身必须进入 graph，producer/consumer 可追踪，否则制造新的隐式 writers
  • A 和 C 不单独触发，总是作为访问隔离的升级路径

高级功能 · WS 调度层（Projection Scheduler）Advanced · WS Scheduler (Projection Scheduler)
⚠ 实现优先级低，Phase 3 之后 这是系统最原创的部分，但依赖前三层全部稳定后才有意义实现。
核心假设：人的 working set 有上限，切换 view 时必须重载的 context 是最昂贵的操作。
展示层不等用户问「给我看什么」，而是主动调度，目标是让每次切换中必须丢弃和重载的部分最小。
人和 AI 走同一套调度逻辑，区别只是输出格式（图形 vs JSON）。Core assumption: Human working set has an upper limit; reloading context on view switches is the most expensive operation.
The presentation layer proactively schedules views rather than waiting for queries, minimizing what must be discarded and reloaded on each switch.
Humans and AI share the same scheduling logic — only the output format differs (visual vs JSON).
当前操作（如：修改 cartTotal） ↓ 计算当前 WS 里已有什么 ↓ 下一步需要什么 ↓ 差集 = 必须重载的部分 调度目标：最小化这个差集（matrix shift 最小） ↓ 自动选择 view + 裁剪局部切片（局部 context 最小） ↓ 人 → 图形化输出 / AI → JSON 切片输出（同一份底层数据）

调度的两个目标

  • matrix shift 最小 — 切换时必须重载的 context 量最小
  • 局部 context 最小 — 单次展示只包含当前操作必要的信息
  • 两者都是 WS thrashing 的对立面
  • 不是「给最多信息」，是「让重载成本最低」

人 vs AI 的差异

  • 调度逻辑：完全一致
  • WS 切片：同一份局部契约数据
  • 人的输出：图形化 view
  • AI 的输出：JSON / 文本切片
  • AI 接入：骨架 interface，不接具体实现
与 RPCM 的对应关系
  • WS thrashing → 切换时重载过多 context，本系统主动压缩
  • WS Compression → 局部切片 = 把大量 graph 压缩成当前操作的最小 chunk
  • Projection Scheduling → 调度层的正式名称，自动展开/折叠 view
  • Adaptive Projection → 根据操作类型（修改/debug/重构）自动切换 view 组合
可借用的开源实现参考
  • Cody 开源快照（sourcegraph/cody-public-snapshot）— RSG + Expand & Refine 算法，context 裁剪和 ranking 逻辑可直接参考，不需要重新设计
  • SCIP（Sourcegraph Code Intelligence Protocol，开源）— 精确代码图，Find References / Definitions，是你 def-use chain 推断（基于 Joern CPG，不需要自己写扫描逻辑）的现成实现基础
  • SonarQube 开源版 — Cognitive Complexity 指标，作为 writers 数量之外的第二健康度维度
  • Joern / Semgrep（均开源）— 静态分析 + pattern matching，可用于 explicit 契约的违反检测
