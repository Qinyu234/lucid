## TASK-002
状态：未完成
做什么：定义 CSF schema（核心数据结构）
输入：PROJECT_CORE.md 第三节
输出：project/core/csf/schema.py

要求：
  CSF 是代码隐式结构的显式化，不是 IR，不是 AST。

  csf_node dict：
  ```python
  {
    "id": str,                    # 唯一 id
    "kind": str,                  # function / class / block / statement / chunk
    "label": str,                 # 人类可读名字
    "source_ref": {               # 对应源码位置
      "path": str,
      "line_start": int,
      "line_end": int,
    },
    "children": list[str],          # 子节点 id 列表（ownership tree）
    "dependency_names": list[str],   # 依赖的函数名/变量名（ast 解析出的字符串）
                                     # 注：不是 node id，是名字字符串
                                     # id → id 的映射在 mutation 标注阶段才建立
    "mutations": list[str],          # 这个 node 赋值的变量名列表
    "mutation_affects": list[str],   # mutation 影响的其他 node id
                                     # 由 TASK-007 annotate_mutations 填充
                                     # v0 只做模块级匹配，不区分 scope（接受假阳性）
    "expansion_state": str,       # "collapsed" | "expanded" | "inline_preview"
    "meta": dict,                 # 附加信息
  }
  ```

  csf dict（顶层）：
  ```python
  {
    "version": str,
    "source_path": str,
    "nodes": dict,         # id → csf_node
    "root_ids": list[str], # 顶层节点 id 列表
    "chunks": list[dict],  # working set chunk 列表
  }
  ```

  chunk dict：
  ```python
  {
    "id": str,
    "node_ids": list[str],  # 属于这个 chunk 的节点
    "label": str,
    "pressure_score": float,  # Recognition Matrix 评分
  }
  ```

  kind 合法值：
    function / class / block / statement / module / chunk

  提供函数：
  ```python
  def empty_csf(source_path: str) -> dict
  def make_node(node_id: str, kind: str, label: str, source_ref: dict) -> dict
  ```

验证：
```
python -c "

csf = empty_csf('test.py')

assert csf['version'] is not None
assert n['kind'] == 'function'
assert n['expansion_state'] == 'collapsed'
print('OK')
"
```

## TASK-003
状态：未完成
做什么：实现 Code → CSF 基础解析（Python）
输入：TASK-002 的 schema
输出：project/core/sync/parser.py

工具选择：
  使用 Python 内置 ast 模块（不自己写 parser）
  ast.parse() 直接给出 AST，ast.NodeVisitor 遍历节点
  不需要安装额外依赖

要求：
  函数签名：def parse(source_path: str) -> dict
  返回 csf dict

  用 ast.NodeVisitor 实现一个 CSFBuilder：

  ```python
  import ast

  class CSFBuilder(ast.NodeVisitor):
      def visit_FunctionDef(self, node): ...   # → kind="function"
      def visit_AsyncFunctionDef(self, node): ...
      def visit_ClassDef(self, node): ...       # → kind="class"
      def visit_If(self, node): ...             # → kind="block", label="if"
      def visit_For(self, node): ...            # → kind="block", label="for"
      def visit_While(self, node): ...          # → kind="block", label="while"
      def visit_Assign(self, node): ...         # → mutations = [target 名]
      def visit_Call(self, node): ...           # → dependency_names 里加被调用函数名
  ```

  source_ref：用 node.lineno 和 node.end_lineno（Python 3.8+ 内置）

  ownership tree：
  - 用 visitor 的调用栈追踪父节点
  - 函数 → 类 → 模块 的层级关系

  dependency_names（同文件，静态可知）：
  - ast.Call 节点：func.id（直接调用）或 func.attr（方法调用）

  mutations：
  - ast.Assign：targets[0] 的名字

  不做：跨文件分析、runtime 分析、类型推断

安装依赖：无（ast 是 Python 内置）

验证：
```
python -c "

print('OK')
"
```

## TASK-004
状态：未完成
做什么：实现继承展开（inheritance expansion）
输入：TASK-003 的 CSF
输出：project/core/expansion/inheritance.py

要求：
  函数签名：def expand_inheritance(csf: dict) -> dict
  返回更新后的 csf（不修改原始，返回副本）

  展开规则：
  - 找到 ClassDef 节点，读取 AST 里的 bases
  - 如果父类在同一文件里定义：
    把父类的 method nodes 作为 "inherited" 虚拟节点附加到子类
    虚拟节点：meta["inherited_from"] = 父类名，meta["virtual"] = True
  - 如果父类不在同一文件：
    创建一个 placeholder 节点：kind="statement"，label="inherited: {父类名}"，
    meta["unresolved_inheritance"] = True

  关键：virtual 节点不改变 source_ref（还是指向父类的原始位置）
  这是 Projection Operation，不改 source，只改 CSF 视图

验证：
```
python -c "
import sys; sys.path.insert(0,'project')
from pathlib import Path
from core.sync.parser import parse
from core.expansion.inheritance import expand_inheritance

p = Path('project/examples/inherit_sample.py')
p.write_text('''
class Animal:
    def speak(self):
        return 'sound'

class Dog(Animal):
    def fetch(self):
        return 'ball'
'''.strip(), encoding='utf-8')

csf = parse(str(p))
csf2 = expand_inheritance(csf)
# Dog 应该能看到 speak 方法
dog_node = next(n for n in csf2['nodes'].values()
                if n['kind']=='class' and n['label']=='Dog')
child_ids = dog_node['children']
child_labels = [csf2['nodes'][c]['label'] for c in child_ids]
assert 'speak' in child_labels, f'inherited speak not found: {child_labels}'
speak_node = next(csf2['nodes'][c] for c in child_ids if csf2['nodes'][c]['label']=='speak')
assert speak_node['meta'].get('inherited_from') == 'Animal'
assert speak_node['meta'].get('virtual') == True
print('OK')
"
```

## TASK-005
状态：未完成
做什么：实现依赖链展开（dependency inline）
输入：TASK-003 的 CSF
输出：project/core/expansion/dependency.py

工具选择：
  同文件依赖：用 TASK-003 的 ast 解析结果（已有 dependency_names 字段）
  跨文件依赖：用 pydeps 生成 module 级依赖图（pip install pydeps）
             v0.1 只做同文件内的 inline，跨文件依赖只标记不展开

要求：
  函数签名：def expand_dependencies(csf: dict, max_span: int = 2) -> dict
  返回更新后的 csf

  同文件 inline 规则：
  - 对每个 function node，查找其 dependency_names 里的函数名
  - 如果该函数名在同一个 csf 的 nodes 里定义（同文件）：
    依赖链深度 <= max_span → inline 展开
    inline 节点：meta["inlined_from"] = 源函数 id，meta["virtual"] = True
  - 依赖链深度 > max_span → 标记 meta["dependency_span_warning"] = True

  跨文件依赖（v0.1 简化）：
  - dependency_names 里找不到同文件定义的函数名
  - 标记 meta["cross_file_dependency"] = True，meta["dependency_label"] = 函数名
  - 不展开，在视图里显示为外部引用标记

  pydeps 集成（可选，如果安装了）：
  ```python
  # 如果 pydeps 可用，补充 module 级依赖信息到 csf meta
  try:
      import pydeps
      # 生成 module 依赖，存入 csf["meta"]["module_deps"]
  except ImportError:
      pass  # 降级：只做同文件分析
  ```

安装依赖：pip install pydeps（可选，不安装也能跑）

验证：
```
python -c "
import sys; sys.path.insert(0,'project')
from pathlib import Path
from core.sync.parser import parse
from core.expansion.dependency import expand_dependencies

p = Path('project/examples/dep_sample.py')

def fetch():
    return get_from_db()

def get_from_db():
    return {'data': 1}

def process():
    data = fetch()
    return data
'''.strip(), encoding='utf-8')

csf = parse(str(p))

                    if n['kind']=='function' and n['label']=='process')
child_ids = process_node['children']
child_labels = [csf2['nodes'][c]['label'] for c in child_ids if c in csf2['nodes']]
print(f'process children: {child_labels}')
# fetch 应该被 inline 展开进 process 的视图

           for c in child_ids), 'no inline expansion found'
print('OK')
"
```

## TASK-006
状态：未完成
做什么：实现 nesting flatten
输入：TASK-003 的 CSF
输出：project/core/expansion/nesting.py

要求：
  函数签名：def flatten_nesting(csf: dict, max_depth: int = 3) -> dict

  规则：
  - 计算每个 block 节点的嵌套深度（在 ownership tree 里距根的距离）
  - 深度 <= max_depth：保持原样
  - 深度 > max_depth：
    标记 meta["nesting_depth"] = 实际深度
    标记 meta["suggest_extraction"] = True
    不自动修改代码，只标记

  同时对 block 节点做 guard chain 识别：
    连续的 if（无 else）→ 标记 meta["guard_chain"] = True
    这表示可以用 early return 重写

验证：
```

def deep_nested():
    if a:
        for b in items:
            if c:
                while d:
                    pass
'''.strip(), encoding='utf-8')

csf = parse(str(p))
csf2 = flatten_nesting(csf, max_depth=3)
# 最深的 block 应该被标记
```

状态：未完成
做什么：实现 mutation 传播标注
输入：TASK-003 的 CSF
输出：project/core/expansion/mutation.py

  # 真正的 scope 分析留待 v1（需要 symtable 或 pylsp 支持）

## TASK-011
状态：未完成
做什么：实现 CSF → Code patch（结构操作写回）
输入：TASK-002，TASK-003
输出：project/core/sync/writer.py

工具选择：
  使用 libcst（pip install libcst）而不是裸 ast
  原因：libcst 是 Concrete Syntax Tree，保留格式/注释/空白
        裸 ast 做 write-through 会破坏代码风格（去掉注释、改变缩进）
        libcst 的 CodeRange 和 CSTTransformer 专门为结构编辑设计

要求：
  安装：pip install libcst

  只实现两个最基础的 Structural Operations：

  1. extract_function（提取函数）
  ```python
  def extract_function(csf: dict, node_ids: list[str],
                       new_name: str, source_path: str) -> dict:
    # 用 libcst 解析源文件
    # 找到 node_ids 对应的 CST 节点
    # 提取成新函数，在原位替换为调用
    # 返回 {"patch": str, "new_csf": dict, "preview": str}
    # patch = unified diff 字符串
    # preview = "这个操作影响：N 处调用点，imports 不变"
  ```

  2. add_early_return（guard chain 转 early return）
  ```python
  def add_early_return(csf: dict, block_node_id: str,
                       source_path: str) -> dict:
    # 用 libcst.CSTTransformer 找到 guard_chain 对应的 if 节点
    # 把嵌套 if 改写为 early return 形式
    # 返回 {"patch": str, "new_csf": dict, "preview": str}
  ```

  libcst 基本用法：
  ```python
  import libcst as cst
  tree = cst.parse_module(source_code)
  # 用 CSTTransformer 做变换
  class EarlyReturnTransformer(cst.CSTTransformer):
      def leave_If(self, original_node, updated_node): ...
  new_tree = tree.visit(EarlyReturnTransformer())
  new_code = new_tree.code  # 保留原始格式
  ```

  patch 生成：用 difflib.unified_diff 对比 old_code 和 new_code

验证：
```
python -c "

from core.sync.writer import add_early_return
import libcst  # 验证 libcst 已安装

p.write_text('''

    if x is None:
        return None
    if x < 0:
        return None
    return x * 2
```

## TASK-012
状态：未完成
做什么：实现 Code → CSF 增量更新（code change → csf patch）
输入：TASK-003
输出：project/core/sync/parser.py（更新，加入 incremental）

工具选择：
  使用 libcst.metadata（pip install libcst）做增量解析
  原因：libcst 的 MetadataWrapper 支持位置信息，方便对比前后变化
  比裸 ast 更稳定处理格式变化（注释变化不会误触发节点更新）

要求：
  函数签名：def incremental_update(old_csf: dict, new_source: str, source_path: str) -> dict

  规则：
  - 用 ast.parse() 重新解析 new_source 得到 new_csf
  - 对比 old_csf 和 new_csf 的 nodes，匹配策略：
    - 按 (kind, label, source_ref.line_start) 做稳定 id 匹配
    - 匹配成功且内容未变 → 保留原有 expansion_state
    - 匹配成功但内容变化 → 更新 source_ref，保留 expansion_state
    - 新增节点 → 加入，expansion_state 默认 "collapsed"
    - 删除节点 → 移除

  关键原则：expansion_state 是用户状态，代码微小变化不应重置它

  libcst 在这里的具体用途：
  - 用 libcst.parse_module 解析，配合 cst.metadata.PositionProvider
  - 获得精确的字符级位置信息，比 ast.lineno 更精确
  - 用于判断"这个节点是否真的变化了"（而不是只是换了行号）

安装依赖：pip install libcst

## TASK-013
状态：未完成
做什么：实现 RPCM（Recognition Projection Control Matrix）Working Set 状态计算
输入：TASK-002 的 CSF schema
输出：project/core/recognition/rpcm.py

说明：
  RPCM 不是压力计，是 Projection Control Model。
  它计算"当前 CSF 的 Working Set 状态"，
  输出供 Projection Scheduler 使用的调度信号。

  WS 类型对应 CSF 的三张图：
    R0 Structural WS → Ownership Tree 的状态
    R1 Execution WS  → Dependency Graph 的状态
    R2 Causal WS     → Mutation Flow Graph 的状态
    R3 System WS     → runtime-opaque 节点的状态（v0 简化：opaque node 计数）

要求：
  函数签名：def compute_ws_state(csf: dict) -> dict

  返回：
  ```python
  {
    "R0": {
      "nesting_depth": int,       # ownership tree 最大深度
      "visible_chunks": int,      # 当前展开的 chunk 数量
      "ws_op": str,               # "stable"|"load"|"compress"（当前对 R0 的操作建议）
    },
    "R1": {
      "dependency_span": int,     # 最长依赖链深度
      "inline_candidates": list[str],  # 建议 inline 展开的 node id
      "ws_op": str,
    },
    "R2": {
      "mutation_fanout": int,     # 最大 mutation_affects 数量
      "high_fanout_nodes": list[str],  # fanout 超限的 node id
      "ws_op": str,
    },
    "R3": {
      "opaque_count": int,        # 系统无法分析的节点数（unresolved_inheritance 等）
      "ws_op": str,
    },
    "thrash_risk": str,           # "none"|"warn"|"critical"
    "thrash_sources": list[str],  # 触发 thrash 的维度列表
  }
  ```

  ws_op 判断规则（启发式阈值，后期校准）：
  ```python
  WS_THRESHOLDS = {
    "R0_nesting_depth":   {"load": 2, "compress": 3},
    "R0_visible_chunks":  {"load": 5, "compress": 7},
    "R1_dependency_span": {"load": 1, "compress": 2},
    "R2_mutation_fanout": {"load": 3, "compress": 5},
    "R3_opaque_count":    {"load": 1, "compress": 3},
  }
  ```

  ws_op 含义：
    "stable"   → 当前 WS 状态正常，不需要调度
    "load"     → 需要加载更多 context（expand）
    "compress" → WS 超限，需要折叠（collapse）

  thrash_risk：
    任何维度 ws_op=="compress" → "warn"
    两个以上维度 ws_op=="compress" → "critical"

验证：
```
python -c "

from core.recognition.rpcm import compute_ws_state, WS_THRESHOLDS

ws = compute_ws_state(csf)
assert 'R0' in ws and 'R1' in ws and 'R2' in ws and 'R3' in ws
assert ws['R0']['ws_op'] in ['stable','load','compress']
assert ws['thrash_risk'] in ['none','warn','critical']
assert 'WS_THRESHOLDS' in dir()  # 阈值是常量，可外部修改
print(f'R0={ws[\"R0\"][\"ws_op\"]} R1={ws[\"R1\"][\"ws_op\"]} R2={ws[\"R2\"][\"ws_op\"]}')
print(f'thrash_risk={ws[\"thrash_risk\"]}')
print('OK')
"
```

## TASK-014
状态：未完成
做什么：实现 Projection Scheduler（基于 RPCM 的 WS 调度）
输入：TASK-013 的 compute_ws_state
输出：project/core/projection/scheduler.py

说明：
  Projection Scheduler 读取 RPCM 的 WS 状态，
  对 CSF 的 expansion_state 做调度决定。
  三种调度操作对应 WS 操作：
    Compress WS → collapse 节点
    Load WS     → expand / inline_preview 节点
    Mutate WS   → 加 overlay 标注（不改结构，加可见性标记）

要求：
  函数签名：def schedule(csf: dict) -> dict
  返回更新了 expansion_state 和 meta 的 csf（副本）

  调度规则：

  R0 compress（nesting 超限）：
    把 nesting_depth > 3 的 block 节点设为 expansion_state="collapsed"
    meta["scheduled_collapse"] = True，meta["reason"] = "R0_compress"

  R1 compress（dependency 超限）：
    把 dependency_span > 2 的依赖链末端节点设为 expansion_state="inline_preview"
    inline_preview = 不折叠，但标记"这是展开的依赖"
    meta["scheduled_inline"] = True，meta["reason"] = "R1_compress"

  R2 compress（mutation fanout 超限）：
    不折叠，而是在 high_fanout_nodes 上加 overlay 标注
    meta["mutation_overlay"] = True，meta["propagation_map"] = mutation_affects 列表

  R0 compress（visible chunks 超限）：
    把 kind="statement" 且 label 长度 < 5 的节点设为 expansion_state="collapsed"
    meta["scheduled_collapse"] = True，meta["reason"] = "R0_low_signal"

  R3 任何状态：
    把 meta["unresolved_inheritance"]=True 的节点加标注
    meta["system_opaque_warning"] = True

验证：
